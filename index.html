Overview

The setup-claude-project.sh script creates a complete project structure optimized for development with Claude Code, Anthropic's AI-powered coding assistant. It establishes documentation, configuration, and workflow patterns that enable effective human-AI collaboration.

What This Script Does

  • Creates standardized project documentation
  • Sets up session continuity files for multi-session work
  • Configures MCP (Model Context Protocol) infrastructure
  • Establishes skill definitions for automated workflows
  • Creates CI/CD pipeline templates
  • Initializes git repository with comprehensive commit

Quick Start

Installation

# Download or copy the script
chmod +x setup-claude-project.sh

# Run with arguments
./setup-claude-project.sh MyProject "A web application for task management"

# Or run interactively
./setup-claude-project.sh

After Setup

# Navigate to your project
cd MyProject

# Start Claude Code
claude

# Claude will automatically load CLAUDE.md and understand your project structure
Tip: Claude Code automatically reads CLAUDE.md at the start of each session. This file contains everything Claude needs to understand your project's conventions, tools, and workflows.

Subagent System

Claude Code uses specialized subagents for different tasks. Each subagent has specific capabilities and tools available to it.

Available Subagents

Type Purpose Tools Best For
Explore Fast codebase exploration Glob, Grep, Read Finding files, understanding structure
Plan Implementation design All tools Architecture decisions, planning
Bash Command execution Bash only Git operations, builds, terminal tasks
general-purpose Complex tasks All tools Multi-step research, anything else

When to Use Each Subagent

Use Explore When:

  • "Where is X implemented?"
  • "How does authentication work?"
  • "Find all files that handle payments"
  • Quick codebase searches

Use Plan When:

  • "How should I implement feature X?"
  • "Design the database schema"
  • Need architectural decisions
  • Multiple implementation approaches

Use Bash When:

  • Git operations (commit, push)
  • Running build commands
  • System administration
  • Terminal tasks

Use General-Purpose When:

  • Complex research
  • Tasks needing multiple tools
  • Anything not fitting above
  • Multi-step investigations

Model Selection

Choose the model based on task complexity and cost sensitivity:

Model Best For Cost Speed
haiku Quick searches, simple tasks Lowest Fastest
sonnet Most coding tasks Medium Balanced
opus Complex reasoning, architecture Highest Slower

Background Tasks

For long-running tasks, use run_in_background: true:

// Agent returns immediately with output_file path
// Check progress with:
// - Read tool to view the output file
// - Bash with `tail -f` to follow output
// - Continue working while agent runs

Parallel Execution

When tasks are independent, Claude can launch multiple agents simultaneously. Each agent gets its own Task tool call, all run concurrently, and results return as they complete.

Planning Mode

Planning mode is a structured workflow for complex implementation tasks that require design before coding.

When to Use Planning Mode

  • New feature implementation - Adding meaningful new functionality
  • Multiple valid approaches - When the task can be solved several ways
  • Architectural decisions - Choosing between patterns or technologies
  • Multi-file changes - Changes affecting more than 2-3 files
  • Unclear requirements - Need to explore before understanding scope

When NOT to Use Planning Mode

  • Single-line or few-line fixes
  • Typo corrections
  • Adding a single function with clear requirements
  • Pure research/exploration tasks (use Explore agent instead)

Planning Mode Workflow

1. User requests complex task ↓ 2. Claude uses EnterPlanMode tool (requires user approval) ↓ 3. Claude explores codebase - Glob, Grep, Read tools - Understand existing patterns ↓ 4. Claude writes plan to file - Implementation steps - Files to modify - Trade-offs considered ↓ 5. If clarification needed: → Use AskUserQuestion tool ↓ 6. Claude uses ExitPlanMode - User reviews plan - User approves or requests changes ↓ 7. After approval: - Execute the plan - Write code, tests, docs

Interactive Clarification

Use the AskUserQuestion tool when you need:

  • User preferences or requirements
  • Clarification on ambiguous instructions
  • Decisions on implementation choices
  • Input on direction to take
Format: 1-4 questions per invocation, 2-4 options per question. Users can always select "Other" for custom input.

Model Context Protocol (MCP)

MCP is Anthropic's open standard for connecting AI assistants to external data sources and tools.

Architecture

┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐ │ AI Assistant │────▶│ MCP Client │────▶│ MCP Server │ │ (Claude Code) │◀────│ (Built-in) │◀────│ (Your tools) │ └─────────────────┘ └─────────────────┘ └─────────────────┘ │ ▼ ┌─────────────────┐ │ External Data │ │ - Databases │ │ - APIs │ │ - Files │ └─────────────────┘

Common MCP Servers

Type Package Use
Filesystem @modelcontextprotocol/server-filesystem File access
PostgreSQL @modelcontextprotocol/server-postgres Database queries
Git @modelcontextprotocol/server-git Git operations
Memory @modelcontextprotocol/server-memory Persistent memory

Configuration

Configure MCP servers in .claude/settings.json:

{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-filesystem",
        "/allowed/path"
      ]
    },
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": {
        "DATABASE_URL": "postgresql://user:pass@host:5432/db"
      }
    }
  }
}

MCP Commands

/mcp                    # List configured servers
/mcp add <server>       # Add a server
/mcp remove <server>    # Remove a server

Development Workflow

The Three Pillars

Every production code change MUST include ALL THREE:

1. Tests

  • Write tests WITH implementation
  • Use pytest for Python
  • Aim for >80% coverage
  • Test success AND failure paths
  • Mock external dependencies

2. Documentation

  • Module-level docstrings
  • Function docstrings (Args, Returns, Raises)
  • Inline comments for complex logic only

3. Git Commits

  • Commit after each logical unit
  • Use conventional commits format
  • Include Co-Authored-By footer
NO EXCEPTIONS. NO SHORTCUTS. Production systems depend on this discipline.

Session Protocol

At Start of Session

  1. Read PROJECT_STATE.md for current status
  2. Read SESSION_LOG.md for recent work and next steps
  3. Check git status for uncommitted changes
  4. Check for running processes

During Session

  • Use TodoWrite for multi-step tasks
  • Commit work frequently
  • Update documentation as you go

At End of Session

  1. Update SESSION_LOG.md with work completed
  2. Commit any uncommitted changes
  3. Note running background jobs
  4. List next steps

Critical Warnings

Shell Script Line Endings

The Write tool adds Windows-style CRLF line endings that break shell scripts on macOS/Linux.

MANDATORY workflow for ALL .sh files:

# 1. Write the script
# 2. IMMEDIATELY fix line endings:
sed -i '' 's/\r$//' script.sh
# 3. Make executable:
chmod +x script.sh
# 4. Test:
./script.sh

Cloud-First Development

Never hardcode local paths. Code must work in cloud environments.

# WRONG
path = '/Users/username/projects/myapp'

# CORRECT
import os
project_root = os.path.dirname(os.path.dirname(os.path.abspath(__file__)))
path = os.environ.get('APP_ROOT', '/app')
